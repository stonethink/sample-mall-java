## 问题清单

### 问题1：缺少用户端展示设计

**问题描述：**
PRD 中只设计了管理后台（admin.html）的促销管理功能，但完全没有涉及用户端（C端）如何展示促销信息。用户在浏览商品、购物车、下单流程中如何感知满减活动？

**影响：**
- 用户无法知晓当前有哪些促销活动
- 无法实时看到已享受的优惠和距离下一档的差额
- 影响活动转化率和用户体验

**建议修改：**
1. 补充商品详情页的促销标签展示设计
2. 补充购物车页面的优惠计算展示
3. 补充订单确认页的优惠明细展示
4. 补充用户端差额提醒的展示方式（如进度条、提示文案等）

**优先级：** P0

---

### 问题2：活动冲突校验规则存在漏洞

**问题描述：**
5.2 节的活动冲突校验规则只考虑了商品范围的交集，但没有考虑**时间维度**的冲突。两个全场通用活动如果时间重叠，即使商品范围相同，也应该视为冲突。

**当前规则缺陷：**
| 场景 | 新活动 | 已有活动 | 当前判定 | 实际应判定 |
|------|--------|----------|----------|------------|
| 场景6 | 全场通用，3.20-4.20 | 全场通用，4.1-4.30 | 冲突 | 冲突（时间重叠）|
| 场景7 | 指定商品A，3.20-4.20 | 指定商品A，4.15-5.15 | 冲突 | 冲突（时间重叠）|

**建议修改：**
冲突判定公式应为：
```
冲突 = (商品范围有交集) AND (时间范围有重叠) AND (两个活动都启用)

时间重叠判定：
!(newEnd < existingStart || newStart > existingEnd)
```

**优先级：** P0

---

### 问题3：订单金额计算逻辑不明确

**问题描述：**
5.4 节优惠计算规则的 Step 1 提到 `originalAmount = Σ(商品单价 × 数量)`，但：
1. 没有说明商品单价是取当前实时价格还是下单时的快照价格
2. 没有说明是否包含运费
3. 没有说明优惠券、积分等其他优惠如何与满减叠加

**影响：**
- 开发时容易产生歧义
- 可能导致金额计算结果与预期不符
- 财务对账困难

**建议修改：**
明确以下规则：
1. 商品单价取下单时刻的商品价格（快照）
2. 满减计算**不包含运费**，运费单独计算
3. 优惠叠加规则：满减优先于优惠券，即 `payAmount = (originalAmount - 满减优惠) - 优惠券抵扣`
4. 增加订单金额计算流程图

**优先级：** P0

---

### 问题4：缺少活动优先级机制

**问题描述：**
当多个活动同时适用于一个订单时（虽然冲突校验应该避免这种情况，但极端情况下仍可能发生），PRD 中只简单说明"选择 reductionAmount 最大的"，这可能导致不合理的结果。

**反例：**
- 活动A：满100减50（优惠力度大但门槛高）
- 活动B：满50减20（优惠力度小但门槛低）
- 订单金额：100元
- 按当前规则：选活动A，优惠50元
- 但可能运营意图是让活动B优先

**建议修改：**
1. 在 Promotion 实体中增加 `priority` 字段（整数，数值越大优先级越高）
2. 修改选择规则：先按优先级排序，同优先级再按优惠金额排序
3. 或增加 `exclusive` 字段标记互斥活动组

**优先级：** P1

---

### 问题5：满减规则缺少"每满减"类型

**问题描述：**
PRD 只设计了阶梯满减（满100减10、满200减30），但没有支持"每满减"类型（每满100减10，可叠加）。这是电商常见的另一种满减形式。

**示例：**
- 阶梯满减：满100减10，满200减30（订单200元只减30，不是20+30）
- 每满减：每满100减10（订单200元减20，300元减30，可叠加）

**建议修改：**
1. 在 FullReductionRule 中增加 `type` 字段：LADDER（阶梯）/ PER_AMOUNT（每满）
2. 或增加 `stackable` 布尔字段表示是否可叠加
3. 修改计算逻辑支持两种模式

**优先级：** P1

---

### 问题6：缺少活动预热机制

**问题描述：**
PRD 中活动状态只有 NOT_STARTED / ACTIVE / ENDED，但没有预热状态。大型促销活动通常需要提前几天开始预热，让用户提前知晓并加购。

**建议修改：**
1. 增加 `preheatTime` 预热开始时间字段
2. 增加 PREHEATING 预热状态：preheatTime <= now < startTime
3. 预热期间活动对用户可见但不可用
4. 管理后台增加预热状态筛选

**优先级：** P2

---

### 问题7：订单接口缺少分页和筛选

**问题描述：**
4.1 节中 `/api/promotions` 接口描述为"查询所有活动（支持状态筛选）"，但没有详细说明：
1. 是否支持分页？分页参数是什么？
2. 支持哪些筛选条件？（状态、时间范围、名称模糊查询等）
3. 排序规则是什么？

**建议修改：**
明确接口参数：
```
GET /api/promotions?page=1&size=20&status=ACTIVE&keyword=春季&sort=createdAt,desc
```

**优先级：** P1

---

### 问题8：缺少数据统计和监控需求

**问题描述：**
PRD 完全没有涉及活动效果的数据统计需求。运营人员需要知道：
1. 活动带来了多少订单？
2. 活动优惠总金额是多少？
3. 活动转化率如何？
4. 各档位的命中分布？

**建议修改：**
增加数据统计接口或管理后台功能：
1. 活动效果概览（订单数、优惠金额、参与用户数）
2. 档位命中分布统计
3. 活动趋势图（按天统计）
4. 导出报表功能

**优先级：** P2

---

### 问题9：技术选型存在兼容性风险

**问题描述：**
2.3 节技术选型中指定使用 Swagger 3.0.0（Springfox），但 Springfox 3.0.0 与 Spring Boot 2.7+ 存在已知的兼容性问题，会导致应用启动失败。

**具体表现：**
```
Failed to start bean 'documentationPluginsBootstrapper'
Cannot invoke "PatternsRequestCondition.getPatterns()" because "this.condition" is null
```

**建议修改：**
1. **方案一（推荐）**：迁移至 SpringDoc OpenAPI
   - 依赖：`springdoc-openapi-ui:1.7.0`
   - Swagger UI 路径变更为：`/swagger-ui/index.html`
2. **方案二**：降级 Spring Boot 版本至 2.6.x
3. **方案三**：添加 `ant_path_matcher` 配置（可能无法完全解决问题）

**优先级：** P0

---

### 问题10：缺少异常处理和降级策略

**问题描述：**
PRD 中没有说明当优惠计算服务异常时如何处理。如果计算服务挂掉，下单流程是否应该阻断？还是允许无优惠下单？

**建议修改：**
1. 明确异常处理策略：
   - 计算服务异常时，允许下单但不享受优惠（降级策略）
   - 记录异常日志，发送告警通知
2. 增加熔断机制设计
3. 增加补偿机制（如事后补发优惠差额）

**优先级：** P1

---

### 问题11：订单扩展字段设计不完整

**问题描述：**
3.4 节 Order 扩展中增加了 `originalAmount`、`promotionDiscount` 等字段，但没有说明：
1. 历史订单（已有数据）如何兼容？新字段默认值是什么？
2. 如果活动规则在订单创建后被修改，历史订单的金额是否应该保持不变？
3. 是否需要记录当时使用的规则快照？

**建议修改：**
1. 历史订单的 `originalAmount` = `totalAmount`，`promotionDiscount` = 0
2. 订单创建时记录规则快照（如 `ruleSnapshot` JSON 字段）
3. 明确订单金额不允许因规则变更而重新计算

**优先级：** P1

---

### 问题12：管理后台权限控制未提及

**问题描述：**
PRD 中管理后台（admin.html）增加了促销管理功能，但没有说明权限控制。是否所有管理员都能创建/修改活动？是否需要审批流程？

**建议修改：**
1. 明确促销管理的权限点（如 `PROMOTION_READ`、`PROMOTION_WRITE`）
2. 说明是否需要审批流程（如活动创建后需审批才能启用）
3. 增加操作日志记录（谁、何时、做了什么操作）

**优先级：** P2
