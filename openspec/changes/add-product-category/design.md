## Context

当前商城系统使用 JSON 文件存储商品和订单数据，商品实体缺少分类字段。需要引入商品分类体系来支持商品的层级组织和筛选展示。

项目技术栈：Spring Boot 2.7+、SpringDoc OpenAPI、内存数据存储（JSON 文件）。

## Goals / Non-Goals

**Goals:**
- 支持多级分类（父子层级），至少支持 3 级深度
- 分类 CRUD 操作完整可用
- 商品可关联多个分类
- 商品列表支持按分类筛选
- 管理后台提供分类管理界面

**Non-Goals:**
- 分类权限控制（所有管理员可操作所有分类）
- 分类排序权重功能
- 分类图片/icon 上传
- 数据库迁移（继续使用 JSON 存储）

## Decisions

### 1. 数据存储方式：JSON 文件
**选择：** 继续使用 JSON 文件存储分类数据，与现有商品/订单存储保持一致。
**理由：**
- 保持架构一致性，无需引入数据库
- 简化部署，无外部依赖
- 分类数据量预计较小，JSON 足够

**替代方案：** 引入 H2/嵌入式数据库 → 增加复杂度，收益有限

### 2. 分类层级实现：父级引用模式
**选择：** Category 实体包含 parentId 字段指向父分类
**理由：**
- 实现简单，适合 JSON 存储
- 支持无限层级
- 查询子分类方便

**替代方案：** 闭包表/路径枚举 → 过于复杂，不适合 JSON 场景

### 3. 商品-分类关联：多对多关系
**选择：** 商品包含 categoryIds 列表，关联分类 ID
**理由：**
- 一个商品可属于多个分类（如：手机 → 数码 + 通讯）
- JSON 中直接存储 ID 列表简单高效

### 4. 分类删除策略：禁止删除有子分类的分类
**选择：** 删除分类时检查是否有子分类，有则禁止删除
**理由：**
- 防止误删导致数据孤儿
- 强制先删除子分类，保持数据完整性

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| JSON 文件并发写入冲突 | 使用同步方法操作文件，单实例部署 |
| 分类层级过深影响性能 | 限制最大层级为 5 级，前端展示限制 |
| 分类数据量大时查询慢 | 当前场景数据量小，如需优化可引入缓存 |

## Migration Plan

1. 新增 Category.java 实体和 categories.json 数据文件
2. 修改 Product.java 添加 categoryIds 字段
3. 初始化默认分类数据（如：数码、服装、食品等）
4. 更新 admin.html 添加分类管理页面
5. 验证商品-分类关联功能正常
